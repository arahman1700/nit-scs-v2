
  <action>
**Problem:** The first two migrations (`0000_baseline` and `0001_add_check_constraints`) use non-timestamp naming. Later migrations use proper timestamp format (`20260312000000_*`). The `prisma migrate deploy` command expects consistent timestamp ordering. Having `0000_` and `0001_` prefixes causes the migrations to sort lexicographically before the timestamp migrations, which is correct, but `prisma migrate deploy` may report issues with mixed formats in CI/CD pipelines.

**Strategy:** Rename the two non-standard migrations to timestamp format. This is safe because:
- These migrations have already been applied to all existing databases (they are the initial baseline)
- The Prisma `_prisma_migrations` table tracks migrations by directory name
- We need to update the migration names in the `_prisma_migrations` table for existing databases via the start.sh script

**Steps:**

1. Rename `0000_baseline` to `20260101000000_baseline`:
   ```bash
   mv packages/backend/prisma/migrations/0000_baseline packages/backend/prisma/migrations/20260101000000_baseline
   ```

2. Rename `0001_add_check_constraints` to `20260101000001_add_check_constraints`:
   ```bash
   mv packages/backend/prisma/migrations/0001_add_check_constraints packages/backend/prisma/migrations/20260101000001_add_check_constraints
   ```

3. Update `packages/backend/start.sh` to handle the migration rename for existing databases. Add BEFORE the `prisma migrate deploy` call:
   ```bash
   echo "==> Renaming legacy migrations in _prisma_migrations table (if needed)..."
   npx prisma db execute --stdin <<'SQL' 2>/dev/null || true
   UPDATE "_prisma_migrations"
   SET migration_name = '20260101000000_baseline'
   WHERE migration_name = '0000_baseline';
   UPDATE "_prisma_migrations"
   SET migration_name = '20260101000001_add_check_constraints'
   WHERE migration_name = '0001_add_check_constraints';
   SQL
   ```

   The `|| true` ensures this is non-fatal (if the table does not exist yet, e.g., fresh database, the command fails harmlessly).

4. Verify the final migration directory listing is all timestamp-prefixed:
   ```
   20260101000000_baseline/
   20260101000001_add_check_constraints/
   20260312000000_p5_oracle_indexes_matviews/
   20260313000000_add_version_columns/
   20260313100000_seed_email_templates/
   20260313200000_add_report_sharing/
   20260313300000_add_po_mirror_tables/
   migration_lock.toml
   ```

5. Verify `prisma migrate deploy` works by running a dry check:
   ```bash
   cd packages/backend && npx prisma migrate status
   ```

Do NOT create a new migration SQL file -- this is purely a directory rename + tracking table update for existing databases.
