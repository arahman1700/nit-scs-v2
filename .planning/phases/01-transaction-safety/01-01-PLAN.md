    **Step 2: Collect alerts in batch functions and publish post-commit**

    ```typescript

      const alerts: LowStockAlert[] = [];
      for (const params of items) {
        // ... existing mutation logic ...
        const alert = await checkLowStockAlert(tx, itemId, warehouseId);
        if (alert) alerts.push(alert);
      }
      return alerts;
    };
    ```

    After the transaction resolves, publish collected alerts:

    ```typescript
    let alerts: LowStockAlert[];
    if (externalTx) {
      alerts = await run(externalTx);
      // When using externalTx, return alerts to caller for post-commit publishing
      // Store on a module-level pending alerts array that the outer tx owner can flush
    } else {
      alerts = await prisma.$transaction(run);
      // Standalone tx -- safe to publish now

        eventBus.publish(alert);
      }
    }
    ```

    **Important nuance for `externalTx` path:** When called with an `externalTx`, the function is inside someone else's transaction. It cannot publish post-commit because it doesn't control the commit. In this case, the alerts should be returned to the caller. However, to minimize API surface change, use a simpler approach:

    Change the batch functions to return the alerts array alongside their existing return values. For functions that return void (addStockBatch, deductStockBatch, releaseReservationBatch), change the return type to `Promise<{ alerts: LowStockAlert[] }>`. For functions that already return values (reserveStockBatch, consumeReservationBatch), add an `alerts` field to the return.

    Then in the callers (grn.service.ts store(), stock-transfer.service.ts ship()/receive(), mrn.service.ts complete(), etc.), collect the alerts and publish after their $transaction commits. Since these callers already correctly publish events post-commit, adding alert publishing there is natural.

    **Simpler alternative (recommended):** Since ALL existing callers already wrap their stock operations in $transaction and publish events post-commit, keep the batch function signatures unchanged. Instead:

    1. Make `checkLowStockAlert` return the alert data (not publish it)
    2. Collect alerts inside the `run` closure
    3. In the standalone path (no externalTx), publish after `prisma.$transaction(run)` resolves
    4. In the externalTx path, the callers already handle post-commit -- but the alerts from `checkLowStockAlert` are currently lost. To fix: store alerts in a closure-scoped array and publish them in a `.then()` appended to the transaction by the callers.

    **Simplest correct approach:** Since `checkLowStockAlert` is private and only called inside batch functions, and batch functions are always called inside transactions owned by document services:

    1. `checkLowStockAlert` returns alert data (not void)
    2. Each batch function collects alerts but does NOT publish them
    3. Each batch function's return value includes `alerts: LowStockAlert[]`
    4. Document service callers (already confirmed correct: grn store, mrn complete, WT ship/receive) publish alerts after their `$transaction` commits

    For the standalone path (batch function called without externalTx), the batch function's own `$transaction` wrapper publishes the alerts after commit.
